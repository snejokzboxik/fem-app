# README для тестировщика

Этот документ предназначен для студента-тестировщика, который немного знаком с
Python, но не обязан глубоко разбираться в физике и численных методах проекта.

## Что это за проект

`charged_particle_trap` — учебный Python-прототип для моделирования движения
одной заряженной микро- или наночастицы в электрическом поле ловушки.

В проекте есть локальный Streamlit-интерфейс на русском языке. Через него можно:

- загрузить или нарисовать геометрию электродов;
- назначить ролям электродов `RF`, `DC`, `GND` или `CUSTOM`;
- посчитать поле простой поверхностной модели через Poisson-kernel;
- импортировать поле или потенциал из `.npz`;
- запустить движение частицы;
- посмотреть траекторию, графики, псевдопотенциал и диагностику Матье;
- выполнить небольшой подбор RF-параметров;
- сохранить результаты, конфиг эксперимента, отчёт и ZIP-пакет.

## Что нужно проверять

Тестировщик проверяет:

- запускается ли проект;
- проходит ли `pytest`;
- открывается ли Streamlit-приложение;
- нет ли красных ошибок сразу после запуска;
- работают ли основные сценарии интерфейса;
- нажимаются ли кнопки, которые выглядят нажимаемыми;
- не теряются ли введённые значения;
- понятны ли предупреждения и ошибки;
- работают ли загрузка, сохранение и экспорт;
- появляются ли ожидаемые графики, таблицы и файлы;
- не зависает ли приложение на маленьких демонстрационных расчётах;
- можно ли воспроизвести проблему по шагам.

## Что тестировщик НЕ обязан проверять

Тестировщик не обязан:

- доказывать физическую корректность модели ловушки;
- проверять строгую математическую корректность FEM или Poisson-kernel solver;
- валидировать реальные лабораторные параметры;
- проверять точность диаграммы Матье для произвольной геометрии;
- оптимизировать скорость расчёта;
- исправлять код.

Важно: тестировщик проверяет интерфейс, падения, очевидно странное поведение,
непонятные кнопки, экспорт, загрузку/сохранение и базовый путь расчёта.
Тестировщик не обязан доказывать физическую корректность модели ловушки.

## Как установить зависимости

Откройте Windows PowerShell в папке проекта:

```powershell
cd "C:\Users\arosl\Desktop\projects py\3d fem\charged_particle_trap"
```

Если виртуальное окружение уже создано, используйте его:

```powershell
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

Если запускаете проект в другом окружении:

```powershell
pip install -r requirements.txt
```

Если при запуске canvas-редактора появляется ошибка про
`streamlit-drawable-canvas`, снова выполните установку зависимостей:

```powershell
pip install -r requirements.txt
```

## Как запустить автоматические тесты

Рекомендуемая команда для этой папки проекта:

```powershell
.\.venv\Scripts\python.exe -m pytest
```

Универсальная команда:

```powershell
python -m pytest
```

Хороший результат — все тесты прошли, например `70 passed`. Предупреждения
можно записать в отчёт, но они не всегда являются багом.

## Как запустить Streamlit

Рекомендуемая команда:

```powershell
.\.venv\Scripts\streamlit.exe run app.py
```

Универсальная команда:

```powershell
streamlit run app.py
```

Обычно браузер откроется автоматически. Если не открылся, скопируйте локальный
URL из консоли Streamlit, например `http://localhost:8501`, и вставьте его в
браузер вручную.

## Какие скриншоты прикладывать

К баг-репорту полезно приложить:

- весь экран с красной ошибкой Streamlit;
- область интерфейса, где кнопка или форма ведёт себя странно;
- настройки sidebar или карточек, которые привели к проблеме;
- график или таблицу, если результат выглядит неправильным;
- список скачанных файлов, если проблема связана с экспортом;
- текст ошибки из консоли PowerShell.

Скриншоты лучше называть понятно:

```text
BUG-003_function_expression_error.png
BUG-007_export_zip_empty.png
BUG-012_canvas_design_reload.png
```

## Как оформлять баги

Используйте шаблон:

```text
docs/qa/BUG_REPORT_TEMPLATE.md
```

Хороший баг-репорт отвечает на вопросы:

- где произошла проблема;
- что вы сделали по шагам;
- что ожидали увидеть;
- что произошло на самом деле;
- можно ли повторить баг;
- есть ли скриншот или текст ошибки;
- насколько проблема серьёзная.

Плохой баг-репорт:

```text
Не работает поле.
```

Хороший баг-репорт:

```text
В режиме "Электроды функциями" выражение "__import__(\"os\")" отклоняется,
но текст ошибки на английском и не объясняет, что именно запрещено.
Шаги: ...
Ожидание: понятное предупреждение на русском.
Факт: ...
```

## Как написать финальный отчёт

После прохождения чеклиста используйте:

```text
docs/qa/TEST_REPORT_TEMPLATE.md
```

В финальном отчёте важно указать:

- сколько проверок выполнено;
- сколько прошло и сколько не прошло;
- какие баги самые важные;
- какие места интерфейса самые непонятные;
- какие скриншоты приложены;
- какая версия Windows/Python использовалась.

## Главный принцип

Если приложение падает, зависает, показывает непонятную красную ошибку,
не сохраняет файл, не загружает файл обратно или вводит пользователя в
заблуждение, это нужно записать как баг или UX-замечание.

Если результат физически кажется странным, но интерфейс работает, запишите это
как наблюдение. Не нужно самостоятельно доказывать, что физическая модель
неверна.
